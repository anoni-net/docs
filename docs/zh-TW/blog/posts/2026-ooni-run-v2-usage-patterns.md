# 我們普查了 336 條 OONI Run v2 清單：被低估的高槓桿用法

![OONI Run v2 使用樣態普查](https://assets.anoni.net/docs/ooni-run-v2-header.png){style="border-radius: 10px;box-shadow:1px 1px 0.6rem #00aeff;"}

全世界的 OONI Run v2 累積跑出 1,417 萬筆檢測，其中 3 條清單就佔了 72%。佔比最高的幾條清單用了同一種作法，每條只針對一個特定的審查或封鎖現象，再用一個持續運行的測量後端定期執行、長期累積資料。我們對全部 Run v2 做了一次普查，量出集中的程度，也整理出這套作法對想做在地連線觀測的社群有什麼參考價值。

<!-- more -->

## 我們做了什麼普查

OONI Run v2 的 link 編號是流水號，從 `10000` 起跳，`10000` 以下都回 404。順著編號往後走，到 2026-06-01 為止最大編號是 `10336`。我們抓下全部存活的 336 條 link 的 metadata，再用 OONI 的 aggregation API[^1] 逐條撈出每條 link 的生涯檢測量。查詢方法、原始資料與腳本都已公開在 [ooni-run-v2-census](https://github.com/anoni-net/ooni-run-v2-census){target="_blank"}，任何人都能重跑驗證。完整步驟放在文末的方法論章節。

## 母體有多大：337 條，兩年

OONI Run v2 從 2024-04-24 第一條 link 上線到現在約 26 個月，全世界總共只建立過 337 條 link（其中 1 條已被刪除，現存 336 條）。建立數在 2024 年 10 到 12 月衝高，之後穩定在每月 10 到 19 條之間。157 個不重複作者參與，332 條走 web_connectivity，幾乎清一色在測網站可達性。

## 使用量極度集中：Gini 0.981

把 336 條 link 的生涯檢測量加起來是 14,171,627 筆。總量不算少，問題在分佈極不平均。以檢測量計算的 Gini 係數是 0.981，幾乎逼近完全不均的 1.0。

!!! note "為什麼用 Gini 係數"

    Gini 係數原本用來衡量所得分配的不平均，數值介於 0 到 1，把整個分佈的集中程度濃縮成一個數字。0 代表完全平均，每條 link 跑出一樣多的檢測。1 代表完全集中，所有檢測都來自同一條 link。

    用在這裡很合適，因為每條 link 的檢測量都是非負的數量，要回答的問題剛好是全網檢測量有多集中在少數 link 上。Gini 在數學上等於 Lorenz 曲線與對角線之間的面積（再除以對角線下方的總面積），所以下面那張 Lorenz 曲線越貼著底邊，Gini 就越接近 1。0.981 代表分佈已經很接近「少數 link 吃下幾乎全部檢測」的極端。

    單一數字看不出是哪幾條 link、長尾長什麼樣，所以這裡同時附上 Lorenz 曲線與分級分佈兩張圖，把集中的形狀也攤出來。

## anoni.net 的座標：社群自己就是一組對照實驗

anoni.net 社群先後建了兩條 Run v2 link，剛好示範前面提到的兩種樣態，兩條都攤開，不只挑表現好的那條。

第一條編號 `10238`，建於 2025-09-15，收了 7 個 input（anoni.net 與 im、matrix、form、pad、search、cinny 等自家服務），生涯只跑了 14 筆。它落在「建了沒持續跑」的多數人那一類。
第二條編號 `10328`，建於 2026-05-15，清單擴成 10 個 input，這次配上自營節點持續跑。半個月就累積 4,659 筆，在 336 條 link 裡排到第 39 名，進到前 12%。

同一個社群、同一份 anoni.net 自家服務清單雛形，兩條 link 的差別只在有沒有持續運行的後端。另一面同樣不迴避，`10328` 佔全網比重仍然趨近 0%。在如此長尾的分佈裡，光是持續跑半個月就贏過 297 條 link，可見「持續運行」這個門檻被多數人錯過。從 `10238` 到 `10328`，就是社群自己跨過門檻的紀錄。

## 對社群與 OONI 的意義

對想做在地連線觀測的社群，普查結果指出一個明確方向。Run v2 的價值來自「一份針對特定現象的策展清單，加上一個持續運行的測量後端」，兩者缺一就會掉進長尾。多開幾條 link 本身不會帶來資料。

同一套模式可以套用在很多題目上。文章前面的 DW、conexionsegura、vesinfiltro，分別是國際媒體可達性、體育盜版封鎖、在地封鎖三種已經在運轉的例子。想開始的人可以從幾個方向發想。

- **選舉觀察**：從選前到選後，持續追蹤新聞媒體、候選人與選務機關網站在特定地區的可達性，把投票期間的封鎖或干擾記錄下來，給選舉觀察員與研究者一份可佐證的時間序列。
- **重大事件即時觀測**：抗議、罷工、災害或斷網期間，持續測獨立媒體、即時通訊服務（Signal、Telegram）與求助資訊網站，第一時間看出哪些被擋、什麼時候被擋。
- **特定族群與議題**：LGBTQ+、宗教少數、女性權益、人權組織這類在某些地區容易被封鎖的網站，做長期縱貫觀測，累積可被研究引用的證據。
- **規避工具可用性**：追蹤 VPN、Tor、橋接與其他 circumvention 工具的官網與下載點，在某地是否還連得上、何時開始被擋，對協助當地人維持連線的團體特別實用。
- **跨境媒體與離散社群**：離散社群依賴的境外媒體（對正體中文使用者例如 RFA、端傳媒），觀測它們在不同 ASN 的可達性差異與長期變化。
- **平台級封鎖事件**：某個 app 或服務在某國突然被擋時，快速建一條清單跨 ASN 觀測，量出封鎖的範圍與起訖時間，把單一事件變成可分析的資料。

挑一個你最關心的題目，列出十來個相關網站，再讓一個節點每天跑，幾週後就有自己的可達性時間序列可以分析。
對 OONI 與做資料分析的夥伴，aggregation API 支援用 `ooni_run_link_id` 篩選，針對 Run v2 生態本身做 meta 分析因此變得可行，本篇報告即依此產出。
anoni.net 社群把這套做法整理在 [issue #78](https://github.com/anoni-net/docs/issues/78){target="_blank"} 的測量模式提案裡，徵詢仍然開放，歡迎社群與 OONI 團隊一起討論。

!!! info "issue #78 的測量模式提案摘要"

    issue #78 提出一套 civic-internet-watch 測量模式，把系統拆成兩層。

    - **名單治理**：社群自己維護一份公民團體網站名單，single source of truth 放在社群的 git repo，收錄標準、徵詢狀態與敏感分級都由社群依在地脈絡決定，git 歷史本身就是清單治理紀錄。
    - **測量與資料**：用標準 web_connectivity 方法，從名單產生一條 OONI Run v2 連結來檢測。志工手機與自營節點消費同一條連結、帶同一個 Run Link ID，結果全部進 OONI 公開資料集，可用 OONI API 與 Explorer 查詢、研究員可引用，沒有資料鎖定。

    名單治理不去更動共享的 Citizen Lab `tw.csv`，避免反覆改動造成長期測量時間序列斷層。`tw.csv` 從必要路徑降為可選的 additive 回饋，成熟非敏感的組織才以只新增的方式回饋換取全平台基線，敏感組織只留在社群自管名單與 Run v2，不進公開的 PR。

## 方法論與資料

普查的每一步都可重現，完整腳本與原始資料公開在 [ooni-run-v2-census](https://github.com/anoni-net/ooni-run-v2-census){target="_blank"}。核心只有兩個查詢。

```bash
# 1) 枚舉 link metadata（流水號從 10000 起，10000 以下為 404）
curl -s "https://api.ooni.org/api/v2/oonirun/links/<ID>"

# 2) 逐條撈該 link 的生涯檢測量
curl -s "https://api.ooni.org/api/v1/aggregation?ooni_run_link_id=<ID>&since=2024-01-01&until=2026-06-30"
# 回傳 result.measurement_count / anomaly_count / ok_count / failure_count
```

解讀以上數字前，有幾個前提要先說明。

- 檢測量來自 aggregation API 的 `measurement_count`，是 2026-06-01 當下的查詢結果，之後還會繼續長。
- 生涯查詢用 `since=2024-01-01` 到 `until=2026-06-30` 涵蓋 Run v2 的整段歷史。
- 檢測量為 0 的 link 可能包含測試或草稿用途，不全等於「失敗」。
- 一條 link 可以被任何國家的 probe 跑，使用情境分群是按作者與清單主題推斷，不是 OONI 的官方標註。
- 完整揭露：社群自己的兩條 link（`10238`、`10328`）都列進分析與討論，沒有只挑表現好的 `10328`。
